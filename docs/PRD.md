## 1. 產品概述

### 1.1 產品定位

ERP 全家桶是一套以 SaaS 模式提供的企業資源規劃平台，採用多租戶架構（Multi-tenant Architecture），為中小企業提供一站式的數位化管理解決方案。平台整合了企業日常營運所需的 11 個核心業務模組，涵蓋銷售、採購、庫存、生產、財務、人力資源、客戶關係管理、品質管理、商業智能、流程管理及 POS 銷售等完整功能。

### 1.2 產品願景

透過統一的資料平台與標準化流程，消除企業各部門之間的資訊孤島，實現跨部門的即時資料共享與協作，協助企業做出更精準的決策、提升營運效率。

### 1.3 核心價值主張

- **一站式整合**: 單一平台涵蓋所有核心業務流程，無需多套系統切換
- **多租戶 SaaS**: 彈性訂閱制，按需付費，快速部署上線
- **資料安全隔離**: 各租戶資料儲存在獨立資料庫 Schema，確保資料完全隔離
- **角色型存取控制**: 細粒度的 RBAC 權限管理，確保人員只能存取授權資源
- **企業級安全**: 雙因素驗證（2FA）、帳號鎖定機制、bcrypt 加密保護

### 1.4 適用規模

- **主要市場**: 員工 10 至 500 人之中小企業
- **次要市場**: 需要模組化 ERP 解決方案之成長型企業

## 2. 目標用戶

### 2.1 用戶角色定義

#### 系統管理員 (System Administrator)
- **身份**: 平台層級的超級管理員
- **職責**: 管理所有租戶、系統備份、平台維護
- **技術程度**: 中至高，熟悉 IT 系統管理
- **痛點**: 需要跨租戶監控、批量管理、緊急備份能力
- **需求**: 統一的管理後台、完整的系統控制權限

#### 租戶管理員 (Tenant Administrator)
- **身份**: 企業內部的 ERP 系統負責人
- **職責**: 管理企業內部用戶帳號、設定角色權限、監控業務數據
- **技術程度**: 中等，熟悉業務流程
- **痛點**: 用戶帳號管理繁瑣、人員異動頻繁需快速調整權限
- **需求**: 直觀的用戶管理介面、清晰的角色設定、審計日誌

#### 部門主管 (Department Manager)
- **身份**: 各業務部門的主管
- **職責**: 審核業務流程、查看部門報表、管理下屬工作
- **技術程度**: 低至中等，非技術背景
- **痛點**: 資料散落各處、報表製作耗時、無法即時掌握部門狀況
- **需求**: 直觀的儀表板、一鍵報表匯出、行動裝置可用性

#### 一般員工 (General User)
- **身份**: 日常使用 ERP 系統的業務人員
- **職責**: 輸入業務資料、追蹤訂單狀態、處理日常業務事務
- **技術程度**: 低至中等
- **痛點**: 系統操作複雜、學習成本高、重複性資料輸入
- **需求**: 簡潔易用的操作介面、快速資料輸入、即時狀態更新

### 2.2 用戶旅程

**典型使用場景 — 銷售訂單流程**:
1. 銷售人員建立客戶資料 (CRM)
2. 銷售人員建立報價單並轉換為銷售訂單 (銷售管理)
3. 系統自動檢查庫存可用量 (庫存管理)
4. 若庫存不足，觸發採購申請 (採購管理)
5. 生產排程根據訂單自動規劃 (生產管理)
6. 訂單出貨後自動建立應收帳款 (財務管理)
7. 管理者在儀表板即時查看訂單 KPI (商業智能)

## 3. 核心功能模組

### 3.1 儀表板 (Dashboard)

**功能描述**: 提供企業整體業務狀況的即時概覽，整合跨模組 KPI 數據。

**主要功能**:
- 跨模組 KPI 卡片展示（銷售額、採購金額、庫存量、待處理訂單數）
- 業務趨勢圖表（月度銷售走勢、收支對比）
- 待辦事項提醒（待審核訂單、低庫存警示、待付款帳款）
- 快速導航至各功能模組
- 資料即時更新，無需手動刷新

**接受標準**:
- 儀表板載入時間不超過 2 秒
- 所有 KPI 數據在 5 分鐘內自動更新
- 支援自定義小工具排列（未來版本）

---

### 3.2 採購管理 (Procurement Management)

**功能描述**: 管理企業採購流程，從需求申請到付款結算的完整採購循環。

**主要功能**:
- **供應商管理**: 供應商基本資料、評級、聯絡人管理
- **採購申請**: 部門採購需求提交、上級審核流程
- **採購訂單**: 建立 PO、發送給供應商、追蹤交貨狀態
- **收貨管理**: 到貨確認、數量核對、品質初驗
- **應付帳款**: 供應商發票登錄、付款排程、付款記錄

**業務規則**:
- 採購金額超過閾值需多層審核
- 供應商評級影響優先採購排序
- 庫存低於安全庫存量時自動觸發採購建議

---

### 3.3 銷售管理 (Sales Management)

**功能描述**: 管理完整的銷售流程，從報價到收款的端對端銷售循環。

**主要功能**:
- **客戶管理**: 客戶基本資料、信用額度、聯絡人
- **報價單**: 建立報價、版本管理、客戶確認
- **銷售訂單**: 訂單建立、庫存確認、出貨安排
- **出貨管理**: 揀貨單、裝箱單、出貨確認
- **應收帳款**: 發票建立、收款追蹤、逾期提醒

**業務規則**:
- 客戶信用額度超過時需主管審核
- 訂單確認後自動扣減可用庫存
- 支援多幣別報價與匯率換算

---

### 3.4 庫存管理 (Inventory Management)

**功能描述**: 即時追蹤貨品庫存狀況，管理倉庫作業與庫存異動。

**主要功能**:
- **商品管理**: 品號設定、規格、單位、條碼
- **倉庫管理**: 多倉庫設定、儲位管理
- **庫存查詢**: 即時庫存量、可用量、在途量
- **庫存異動**: 入庫、出庫、調撥、盤點調整
- **安全庫存**: 安全庫存量設定、低庫存警示
- **庫存盤點**: 定期盤點作業、差異分析

**業務規則**:
- 先進先出 (FIFO) 出庫原則
- 支援序號追蹤與批次管理
- 庫存異動全程記錄不可刪除

---

### 3.5 生產管理 (Manufacturing Management)

**功能描述**: 管理製造流程，從生產計劃到成品入庫的完整生產循環。

**主要功能**:
- **物料清單 (BOM)**: 多層 BOM 建立與管理
- **生產工單**: 工單建立、排程、進度追蹤
- **產能規劃**: 設備與人員產能管理
- **工序管理**: 製程路徑、工時記錄
- **物料需求規劃 (MRP)**: 根據訂單自動計算物料需求
- **成本核算**: 生產成本分析、標準成本 vs 實際成本

**業務規則**:
- 工單物料領料需確認庫存可用量
- 生產完工自動觸發成品入庫
- 生產異常需記錄原因並通知主管

---

### 3.6 財務管理 (Finance Management)

**功能描述**: 提供完整的會計帳務管理，支援企業財務報表編製。

**主要功能**:
- **會計科目**: 科目表建立與管理
- **傳票管理**: 手動傳票、自動傳票（來自各模組）
- **應收帳款**: 帳款管理、收款登錄、帳齡分析
- **應付帳款**: 帳款管理、付款申請、帳齡分析
- **財務報表**: 損益表、資產負債表、現金流量表
- **成本中心**: 部門費用分攤與管控

**業務規則**:
- 傳票借貸必須平衡方可過帳
- 會計期間關帳後不允許修改
- 財務報表數據即時計算

---

### 3.7 人力資源 (Human Resources)

**功能描述**: 管理企業人員相關事務，涵蓋員工資料至薪酬計算。

**主要功能**:
- **員工管理**: 員工基本資料、職位、部門、到職/離職管理
- **組織架構**: 部門樹狀結構、匯報關係
- **考勤管理**: 上下班打卡、請假申請、加班記錄
- **薪酬管理**: 薪資結構設定、薪資計算、薪資條
- **績效管理**: 績效目標設定、定期考核
- **招募管理**: 職缺發布、履歷追蹤、面試安排

**業務規則**:
- 員工資料異動需主管審核
- 薪資計算需與考勤資料自動整合
- 離職員工資料保留不可刪除

---

### 3.8 客戶關係管理 (CRM)

**功能描述**: 管理客戶互動與銷售機會，提升客戶滿意度與留存率。

**主要功能**:
- **聯絡人管理**: 客戶聯絡人資料、互動歷史
- **商機管理**: 銷售漏斗、商機追蹤、預測銷售額
- **活動記錄**: 電話、Email、會議記錄
- **客服工單**: 客訴記錄、處理追蹤、滿意度調查
- **市場行銷**: 行銷活動管理、效益追蹤

**業務規則**:
- 商機逾期未跟進自動提醒負責人
- 客戶重複資料自動合併提示
- 重要客戶標記特殊服務等級

---

### 3.9 品質管理 (Quality Management)

**功能描述**: 建立品質管理體系，確保產品與服務符合標準。

**主要功能**:
- **檢驗計劃**: 進料/製程/出貨檢驗標準設定
- **品質檢驗**: 檢驗記錄、AQL 抽樣計劃
- **不合格品管理**: 不良品記錄、原因分析、處置決定
- **客訴處理**: 客戶抱怨登錄、根因分析、矯正措施
- **品質報告**: 品質趨勢分析、PPM 計算、供應商品質評比

**業務規則**:
- 進料未完成檢驗不允許入庫
- 不合格品需有明確處置記錄
- 品質異常自動通知相關主管

---

### 3.10 商業智能 (Business Intelligence)

**功能描述**: 提供跨模組的資料分析與視覺化報表，支援管理決策。

**主要功能**:
- **預建報表**: 銷售分析、採購分析、庫存分析、財務分析
- **自定義報表**: 拖拉式報表建立工具
- **資料視覺化**: 折線圖、長條圖、圓餅圖、散佈圖
- **KPI 儀表板**: 關鍵指標監控、目標達成率
- **資料匯出**: Excel、CSV、PDF 格式匯出

**業務規則**:
- 報表資料依用戶權限自動過濾
- 大型報表採非同步生成
- 歷史報表快取 24 小時

---

### 3.11 流程管理 (BPM — Business Process Management)

**功能描述**: 提供可視化的業務流程設計與自動化引擎。

**主要功能**:
- **流程設計**: 拖拉式流程圖設計器
- **審核流程**: 多層審核設定、條件分支
- **表單設計**: 自定義審核表單
- **流程監控**: 流程執行狀態、瓶頸分析
- **流程觸發**: 事件觸發、排程觸發、手動觸發

**業務規則**:
- 流程版本管理，舊版本不可修改
- 審核超時自動升級至上一層
- 流程日誌完整記錄所有步驟

### 3.12 POS 銷售 (Point of Sale)

**功能描述**: 提供實體銷售據點的 POS 收銀功能，與庫存及財務模組整合。

**主要功能**:
- **銷售收銀**: 快速商品查詢、條碼掃描、購物車管理
- **付款方式**: 現金、信用卡、電子錢包、發票開立
- **班次管理**: 開收銀機、班次結帳、銷售彙總
- **折扣促銷**: 商品折扣、滿額折扣、會員優惠
- **退換貨**: 退貨處理、換貨、庫存自動回沖
- **多門市管理**: 跨門市庫存查詢、調撥

**業務規則**:
- POS 交易即時更新庫存
- 班次結帳需核對現金差異
- 退貨需核對原始交易單

## 4. 平台管理功能

> 此章節說明 v1.1.0 新增的平台層級管理功能，僅限系統管理員或租戶管理員存取。

### 4.1 用戶管理 (User Management)

**功能描述**: 提供完整的系統用戶帳號生命週期管理，包含帳號建立、角色指派、狀態管理等功能。

#### 4.1.1 用戶帳號管理 (CRUD)

| 功能 | 說明 | 權限需求 |
|------|------|----------|
| 查看用戶列表 | 顯示租戶內所有用戶基本資訊 | `user:users:view` |
| 查看用戶詳情 | 顯示單一用戶完整資訊 | `user:users:view` |
| 新增用戶 | 建立新用戶帳號並指派角色 | `user:users:create` |
| 修改用戶 | 更新用戶資料與角色 | `user:users:update` |
| 刪除用戶 | 刪除用戶帳號（不可恢復） | `user:users:delete` |

**用戶資料欄位**:
- 姓名（必填）
- Email（必填，唯一）
- 密碼（新增時必填，修改時選填）
- 角色（必填）
- 部門（選填）
- 電話（選填）
- 狀態（啟用/停用/鎖定）

#### 4.1.2 角色管理 (RBAC)

系統預設三種用戶角色：

| 角色 | 代碼 | 說明 | 適用範圍 |
|------|------|------|----------|
| 系統管理員 | `super_admin` | 平台最高權限，可管理所有租戶 | 平台層級 |
| 租戶管理員 | `tenant_admin` | 租戶內最高權限，可管理本租戶用戶 | 租戶層級 |
| 一般用戶 | `user` | 依指派權限使用業務模組 | 租戶層級 |

#### 4.1.3 帳號狀態管理

| 狀態 | 說明 | 觸發條件 |
|------|------|----------|
| 啟用 (Active) | 正常可登入使用 | 帳號建立時預設 |
| 停用 (Inactive) | 管理員手動停用，無法登入 | 管理員操作 |
| 鎖定 (Locked) | 密碼連續輸入錯誤自動鎖定 | 系統自動 (5次失敗後) |

**狀態轉換規則**:
- 停用 → 啟用: 需管理員操作
- 鎖定 → 啟用: 需管理員操作，無法自動解鎖
- 啟用 → 鎖定: 系統自動，連續登入失敗 5 次

#### 4.1.4 用戶搜尋與篩選

- 依姓名/Email 關鍵字搜尋
- 依角色篩選
- 依狀態篩選（啟用/停用/鎖定）
- 分頁顯示（每頁 20 筆）

### 4.2 雙因素驗證 (Two-Factor Authentication, 2FA)

**功能描述**: 提供基於 TOTP（Time-based One-Time Password）的雙因素驗證功能，增強帳號安全性，符合 Google Authenticator 標準。

#### 4.2.1 2FA 設定流程

**用戶端設定步驟**:

1. **進入設定頁面**: 用戶登入後，點擊右上角個人頭像 → 選擇「帳號設定」→ 選擇「雙因素驗證」分頁
2. **啟動設定**: 點擊「設定 2FA」按鈕，系統顯示 QR Code 及手動輸入金鑰
3. **掃描 QR Code**: 使用 Google Authenticator 或相容 App 掃描 QR Code，App 會自動新增帳號
4. **驗證啟用**: 輸入 App 顯示的 6 位數驗證碼，點擊「啟用 2FA」確認
5. **完成啟用**: 系統確認後顯示成功訊息，之後每次登入需輸入驗證碼

#### 4.2.2 2FA 登入流程

1. 用戶輸入 Email 與密碼
2. 密碼驗證通過後，若帳號已啟用 2FA，系統跳轉至 2FA 驗證頁面
3. 用戶開啟 Google Authenticator，輸入當前顯示的 6 位數驗證碼
4. 系統驗證通過後完成登入

#### 4.2.3 2FA 停用流程

用戶可自行停用 2FA：進入「帳號設定」→「雙因素驗證」→ 點擊「停用 2FA」→ 輸入當前驗證碼確認。

#### 4.2.4 管理員重設用戶 2FA

**使用時機**: 用戶遺失 Authenticator App 或換手機無法登入時。

**操作流程**:
1. 管理員進入「管理後台」→「用戶管理」
2. 找到該用戶，點擊操作選單
3. 選擇「重設 2FA」
4. 確認操作後，系統清除用戶的 2FA 設定
5. 通知用戶重新登入並重新設定 2FA

**權限需求**: `user:users:update`（租戶管理員以上）

### 4.3 租戶管理 (Tenant Management)

**功能描述**: 提供系統管理員對平台所有租戶的完整管理能力，包含租戶建立、方案管理、狀態控制及資料匯出。

> 此功能**僅限系統管理員 (super_admin)** 使用。

#### 4.3.1 租戶資料管理 (CRUD)

| 功能 | 說明 |
|------|------|
| 查看租戶列表 | 顯示所有租戶基本資訊、狀態、方案 |
| 查看租戶詳情 | 顯示租戶完整資訊、用戶數、建立時間 |
| 新增租戶 | 建立新租戶，自動配置獨立資料庫 Schema |
| 修改租戶 | 更新租戶資料與方案 |
| 刪除租戶 | 移除租戶（需二次確認，不可恢復） |

**租戶資料欄位**:
- 租戶名稱（必填，唯一）
- Schema 名稱（自動生成，格式: `tenant_XXXX`）
- 訂閱方案（基本/標準/企業）
- 狀態（啟用/暫停）
- 聯絡人資訊
- 建立日期

#### 4.3.2 租戶狀態管理

| 操作 | 說明 | 影響 |
|------|------|------|
| 暫停租戶 | 暫時停止租戶服務 | 租戶所有用戶無法登入 |
| 啟用租戶 | 恢復租戶正常服務 | 租戶用戶恢復存取 |

#### 4.3.3 CSV 匯出

系統管理員可匯出所有租戶資料為 CSV 檔案，包含以下欄位：
- 租戶 ID、名稱、Schema、方案、狀態、用戶數、建立時間

**操作**: 點擊租戶列表頁面右上角「匯出 CSV」按鈕，瀏覽器自動下載。

### 4.4 系統備份 (Database Backup)

**功能描述**: 提供系統管理員下載完整資料庫備份的功能，確保資料安全性與災難復原能力。

> 此功能**僅限系統管理員 (super_admin)** 使用。

#### 4.4.1 備份說明

- **備份類型**: PostgreSQL 完整備份（pg_dump 格式）
- **備份範圍**: 所有 Schema（public + 所有租戶 Schema）
- **檔案格式**: `.sql` 純文字 SQL 腳本
- **檔案命名**: `backup_YYYYMMDD_HHmmss.sql`

#### 4.4.2 操作步驟

1. 登入系統管理員帳號
2. 進入「管理後台」→「系統管理」
3. 點擊「下載資料庫備份」
4. 系統於後端執行 pg_dump，完成後自動下載 `.sql` 檔案

#### 4.4.3 安全性注意事項

- 備份檔案包含所有租戶的明文資料，請妥善保管
- 建議下載後立即儲存至安全的離線儲存媒體
- 定期備份頻率建議：每日一次（生產環境）

## 5. 技術需求

### 5.1 多租戶架構

- 採用 **Schema-per-Tenant** 模式，每個租戶擁有獨立的 PostgreSQL Schema
- `public` Schema 儲存平台層級資料（用戶、租戶、角色、權限）
- 每個租戶 Schema 命名格式：`tenant_XXXX`
- 租戶 Schema 隔離確保資料不互相存取

### 5.2 存取控制

- 採用 **RBAC（Role-Based Access Control）** 模型
- 權限格式：`module:resource:action`（例：`sales:orders:create`）
- 支援角色繼承與細粒度權限設定
- API 端點透過 Guards 強制執行權限驗證

### 5.3 身份驗證

- **主要認證**: JWT（JSON Web Token），有效期 15 分鐘
- **令牌刷新**: Refresh Token，有效期 7 天，支援 Rotation 機制
- **雙因素驗證**: TOTP 標準（RFC 6238），相容 Google Authenticator
- **密碼安全**: bcrypt 雜湊，cost factor 12

### 5.4 API 規範

- RESTful API 設計原則
- 統一版本前綴：`/api/v1/`
- 標準 HTTP 狀態碼
- 統一錯誤回應格式
- Swagger/OpenAPI 文件自動生成

### 5.5 系統整合需求

- 各業務模組間需透過 API 或 Service 層整合，不允許跨模組直接存取資料庫
- 前端透過統一 API Client 存取後端，自動處理 Token 刷新
- 支援未來擴充 Webhook 機制（規劃中）

## 6. 非功能需求

### 6.1 安全性 (Security)

| 需求 | 說明 | 標準 |
|------|------|------|
| 密碼強度 | 密碼需符合複雜度要求 | 最少 8 字元，含大小寫字母、數字、特殊字元 |
| 帳號鎖定 | 防止暴力破解 | 連續 5 次失敗後鎖定，需管理員解鎖 |
| 雙因素驗證 | 增強帳號安全性 | 支援 TOTP，建議管理員帳號強制啟用 |
| 密碼加密 | 防止資料庫洩露 | bcrypt 雜湊，不儲存明文密碼 |
| Token 安全 | 防止 Token 盜用 | Refresh Token Rotation，舊 Token 立即失效 |
| 傳輸安全 | 防止中間人攻擊 | 生產環境強制 HTTPS |
| SQL 注入防護 | 防止資料庫攻擊 | 全面使用 Prisma ORM 參數化查詢 |
